// foundation

First, what is authentication?

Authentication is the process of proving your identity to a computer system. Think of it like showing your ID at the door — the system needs to know you are who you say you are before letting you in.

🔐
NTLM
NT LAN Manager — a challenge-response protocol developed by Microsoft in the early 1990s. The server challenges you with a random number; your PC uses your password hash to encrypt it and sends it back.

Think of it as: "Prove you know the secret without saying it out loud."
Legacy Peer-to-peer No 3rd party needed
🎟️
Kerberos
Named after the 3-headed dog of Greek mythology — a ticket-based protocol designed at MIT and used in Windows domains. A trusted authority (the KDC) hands you a "ticket" so you can access services without re-entering your password.

Think of it as: "Get a wristband at the entrance, use it all day."
Modern Domain-based Requires KDC

1 Client → Server Client sends a Negotiate message — "Hello, I want to authenticate using NTLM." NEGOTIATE
2 Server → Client Server replies with a Challenge — a random 8-byte number called a nonce. CHALLENGE
3 Client hashes Client takes the challenge + NT hash of your password and encrypts it using HMAC-MD5. CRYPTO
4 Client → Server Client sends the encrypted result — the Response. Password never travels on the wire. AUTHENTICATE
5 Server verifies Server does the same calculation with the stored hash. If responses match → ✅ Access granted. VERIFY

KDC / DC
(Ticket Office)
🖥️
Service
(Resource)
1 Client → KDC Client sends an AS-REQ — "I am Alice, please give me a ticket." Includes a pre-auth timestamp. AS-REQ
2 KDC → Client KDC verifies Alice's identity and returns an AS-REP: a TGT + session key, encrypted with Alice's key. AS-REP + TGT
3 Client → KDC Client presents the TGT and asks for a Service Ticket for the target server — a TGS-REQ. TGS-REQ
4 KDC → Client KDC returns a Service Ticket (TGS-REP) encrypted with the service's secret — the client can't tamper with it. TGS-REP
5 Client → Service Client presents the Service Ticket as an AP-REQ. No password exchange ever happens with the service. AP-REQ
6 Service verifies Service decrypts the ticket with its own key, reads the identity inside → ✅ Access granted. Optionally sends AP-REP for mutual auth. AP-REP

NTLM vs Kerberos at a glance

Feature 🔐 NTLM 🎟️ Kerberos
Invented ~1993, Microsoft 1988, MIT
Requires Active Directory? ✗ No ✓ Yes (KDC)
Password on wire? ✓ Never (hash-based) ✓ Never (ticket-based)
Mutual Authentication ✗ No ✓ Yes
Single Sign-On (SSO) ✗ Limited ✓ Full SSO
Delegation support ✗ No ✓ Yes
Works without DNS? ✓ Yes ✗ Requires DNS + KDC
Common attack Pass-the-Hash, Relay Pass-the-Ticket, Kerberoasting
Still used today? Fallback only (legacy) Default in Windows domains
